# PM PRD

Use this skill when the user asks for a PRD, requirement document, feature brief, user stories, acceptance criteria, scope, non-goals, or a rewrite of unclear requirements.

## Workflow

1. Identify the product problem, target users, business goal, success metrics, constraints, dependencies, timeline, and decision owner.
2. If key information is missing, start with a short clarification section and state assumptions before drafting.
3. Draft the PRD with clear scope, non-goals, user stories, acceptance criteria, edge cases, analytics, launch plan, and open questions.
4. Separate facts, assumptions, and recommendations.
5. Keep the first draft actionable enough for design and engineering review.

## Output

- 背景与问题
- 目标用户与场景
- 业务目标与成功指标
- 范围与非目标
- 用户故事与验收标准
- 关键流程与边界情况
- 数据埋点与观察指标
- 依赖、风险与降级方案
- 里程碑与下一步
- 假设、证据、待确认项

## Guardrails

- Do not invent company data, internal platform names, or customer quotes.
- If the feature is vague, ask clarifying questions first, then provide a draft based on explicit assumptions.
- Prefer measurable outcomes over feature-completion language.
