# PRD Development

Use this skill when the user needs a complete delivery-grade PRD workflow, not only a document draft: evidence, scope, requirements, review, launch readiness, and post-launch measurement.

Source note: inspired by `deanpeters/Product-Manager-Skills` prd-development at a high level only. The source license is CC BY-NC-SA 4.0, so this skill is not copied or adapted from its text.

## Workflow

1. Confirm PRD purpose: decision, build handoff, alignment, or launch plan.
2. Gather inputs: problem brief, discovery evidence, target users, metrics, constraints, dependencies, design/tech assumptions, and timeline.
3. Define scope:
   - Must ship.
   - Can ship later.
   - Explicit non-goals.
   - Edge cases that change feasibility.
4. Write requirements in delivery language: user stories, acceptance criteria, data needs, permissions, states, failure modes, and operational support.
5. Review with functions: engineering, design, data, support, sales/CS, legal/privacy as needed.
6. Convert review feedback into decisions, open questions, or scope changes.
7. Attach launch and measurement plan before marking ready.

## Output

- PRD readiness checklist
- Complete PRD draft
- Requirement table
- Acceptance criteria
- Analytics and instrumentation plan
- Review plan
- Launch readiness notes
- Open decisions

## Guardrails

- A PRD without success metrics, non-goals, or dependencies is not ready for build handoff.
- Do not bury risky assumptions in prose; list them explicitly.
- If discovery evidence is missing, include a pre-PRD validation step.
