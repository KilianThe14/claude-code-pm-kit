# Problem Definition

Use this skill when the user starts with "we should build", "add a feature", "users want", a stakeholder request, or a fuzzy product idea and needs to define the real problem first.

Source note: inspired by the topic focus of RefoundAI Lenny Skills `problem-definition` (MIT). This version is written for this kit.

## Workflow

1. Translate the request into a problem hypothesis: who struggles, in what context, with what cost.
2. Separate symptoms, root causes, and proposed solutions.
3. Gather evidence: user quotes, behavior data, support/sales signals, competitive pressure, operational cost, or strategic constraint.
4. Test problem quality:
   - Specific user segment.
   - Observable trigger or situation.
   - Meaningful pain or opportunity.
   - Current workaround.
   - Business relevance.
   - Clear success criteria.
5. Explore no-build and low-build options before defaulting to product changes.
6. Produce a problem brief ready for discovery, prioritization, or PRD work.

## Output

- Original request
- Problem statement
- Target users and scenarios
- Evidence and confidence
- Root cause hypotheses
- Current alternatives
- Success criteria
- Solution constraints
- No-build options
- Open questions

## Guardrails

- Do not accept "users asked for X" as the problem.
- Do not write solution requirements until the problem, user, and success measure are explicit.
- Call out when the problem is actually internal process, positioning, onboarding, or support rather than product surface.
