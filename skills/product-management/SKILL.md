# Product Management

Use this skill when the user needs a PM operating system rather than a single artifact: product audit, competitive landscape, gap analysis, GitHub Issue triage, roadmap planning, or turning scattered inputs into a coherent execution plan.

Source note: inspired by the workflow shape of `ooiyeefei/ccc` product-management. This is original guidance for this kit; do not copy source text.

## Workflow

1. Frame the workspace: product area, target users, business objective, decision horizon, constraints, and source inputs.
2. Build the landscape: customer jobs, alternatives, direct competitors, internal capabilities, market expectations, and known gaps.
3. Normalize intake: cluster requests, bugs, research findings, sales asks, support pain, and GitHub Issues into problem themes.
4. Prioritize with a WINNING lens:
   - Win condition: what outcome makes this worth doing.
   - Impact: user/business magnitude.
   - Need evidence: strength of signal.
   - Near-term fit: timing, strategy, and capacity.
   - Investment: effort, dependencies, and opportunity cost.
   - Negatives: risk, reversibility, and second-order effects.
   - Go/no-go: recommendation and next decision.
5. Convert the top themes into roadmap slices, PRD candidates, experiments, or explicit no-build decisions.
6. End with owner, artifact, decision, and follow-up cadence.

## Output

- PM workspace summary
- Landscape and gap map
- Intake clusters
- WINNING prioritization table
- Roadmap recommendation
- PRD or experiment candidates
- Risks, dependencies, and unresolved decisions
- Next actions with owners

## Guardrails

- Do not treat every request as a roadmap item; preserve the problem behind the request.
- Mark source quality for each input: observed behavior, customer quote, metric, stakeholder ask, or assumption.
- If GitHub Issues are referenced but not provided, ask for issue links or exported issue text.
